# 编排方案复用（可选）

**默认不用。** 主 agent 每次自主决策派发方案即可——大多数用户不需要复用机制。本文件只为重复性强的工作流留一个零成本的口子。

## 保存

用户说「保存本次编排」时，把本次的：

- 任务模式（什么类型的任务触发这套编排）
- 派发矩阵（哪一路用什么档位、什么角色、哪个厂商）
- 轮次结构（几轮 review、什么条件下升级/熔断）
- 验证方式（怎么判定"过了"）

写成 `<本skill目录>/playbooks/<名字>.md`。

## 复用

用户说「按方案 `<名字>` 执行」→ 读该文件照做。

**关键：模型 ID 仍取 manifest 的实时值，不用方案里记录的旧 ID。** 方案记录的是「档位 + 厂商 + 角色」这类稳定语义，不是具体型号——型号会漂（见 `channels.md` 末节）。

## 示例结构

```markdown
# playbook: coding-双厂商闭环

## 适用
需要交叉验证的功能实现类任务

## 派发矩阵
| 路 | 角色 | 档位 | 厂商约束 |
|---|---|---|---|
| 1 | 实现 | 中 | 任意 |
| 2 | 审查 | 高 | 必须 ≠ 实现方厂商 |

## 轮次
实现 → 本地跑验证 → 审查 → 修正 → 重跑验证 → 复审，≤3 轮
任一通道连续失败 2 次 → 熔断切同档备选

## 通过判据
审查方给出 PASS 且本地全部测试通过
```

# playbook: 学术/长文本双厂商审查

## 适用
论文/长文本审查（逻辑流、方法学、实验设计等）需要跨厂商核查

## 审查透镜（对材料逐节问）
逻辑流与论证 / 表达清晰度 / 方法学合理性 / 引用恰当性 / 数学与算法正确性 / 实验设计 / 统计分析 / 可复现性

## 派发矩阵（按缺陷深度定 thinking，不按档位；见 SKILL.md「审查任务的思考按缺陷深度开」）
| 检查类型 | 深度 | thinking | 通道 |
|---|---|---|---|
| 单节内硬伤（数值自相矛盾、模板化泄漏） | 浅操作 | 关 | 便宜三家关思考取并集（召回已够，先买第二家厂商，别开思考——收紧阈值反而漏判断型缺陷） |
| 跨节一致性缺陷（结论 vs 数据、方法 vs 样本、单位/基准前后不一） | 深操作 | **必须开**（实测 +21pp，67%→88%） | 开思考跑一家，再换厂商补漏（on 并集可到 100%；深缺陷"多想"与"多家"互补，不互替） |
| 跨节计数核对（"N 种"之类） | — | 不派模型 | 交给脚本核对，不问模型（连开思考也压不上去） |

## 输出骨架
```
## 审查小结

### Critical（必须改）
### High
### Medium
### Low

### Consensus（多家一致的点）
### Divergent（分歧——不仲裁，并列交用户裁决）
- 厂商A：[X]
- 厂商B：[Y]

### Action Items
1. [按严重性排序]
```

## 通过判据
分歧不仲裁——并列呈现交用户裁决（遵守 SKILL.md 执行闭环规则）。**不产出"置信度"评级**（如"两家都同意=高置信"）：共识只对结论生效、会掩盖坏推导，蒸馏/同源收敛下投票近乎无效，详见 SKILL.md。
