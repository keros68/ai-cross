# 在 Qoder 里测 ai-cross（安装 + 两阶段测试 + 记录清单）

Qoder 没有 CLI，所以这是**手动测试**：你在 Qoder 界面里操作，全程录屏或多截图，作为文章素材。

## 一、安装（3 步）

1. 新建（或打开）一个空项目目录，在 Qoder 里打开它。
2. 在项目根新建 `ai-cross/` 目录，把仓库根的 `SKILL.md` 和 `references/` 拷进去，得到 `<项目>/ai-cross/SKILL.md` 和 `<项目>/ai-cross/references/`（或直接把整个仓库 clone 为 `<项目>/ai-cross/`）。
3. 把本目录的 `AGENTS.md` 拷到**项目根**（`<项目>/AGENTS.md`）。Qoder 会自动识别项目根的 AGENTS.md，无需额外配置。

> 若 Qoder 提示用 `.qoder/rules`：把 `AGENTS.md` 内容放进 `.qoder/rules/ai-cross.md`，设为 `Always Apply`，效果一样。

装完新开一个对话，确认 Qoder 已加载（可先问它"你现在遵循哪些项目规则？"，看它有没有提到 ai-cross）。

## 二、阶段 1 —— 派发计划（纸面，一定能测）

测的是：Qoder 当宿主时，能不能①认出自己不是 Claude Code、排除内部通道 ②把交叉审查判给显式不同厂商。**把下面整段粘给 Qoder：**

```
你现在按项目里的 ai-cross skill 工作（先读 ./ai-cross/SKILL.md 和项目根 AGENTS.md）。

场景：你是宿主 agent（Qoder，底层是自动路由的多厂商混合）。用户给一个复合任务：
① 给一批用户反馈做情感分类（纯文本）
② 写一个 Python 统计函数
③ 对②的代码做交叉审查

只输出【派发计划】，不要执行任何命令，逐条注明依据 skill 的哪条规则：
1. 作为 Qoder 宿主，哪些通道可用、哪一类【不可用】？为什么？
2. 子任务①（纯文本分类）走哪个通道？依据哪条规则？
3. 子任务②（写统计函数）走哪个通道？
4. 子任务③交叉审查：你底层这轮可能就是 Claude 或 GPT，为什么不能"再问一次自己"、也不能依赖 Qoder 自动路由？必须显式派给哪一类模型？
5. ②要统计计数，"亲自算"铁律要求你怎么做？
```

**期望**：它说内部 subagent 不可用（走外部命令）；①纯文本→裸 API；②写代码→外部通道；③必须显式换厂商（非当前底层厂商）；⑤统计要写代码跑 + 编排方独立核验。

## 三、阶段 2 —— 真实跨厂商派发（取决于 Qoder 放不放行命令）

测的是：Qoder 能不能真的 shell 出去、派给一个明确不同的厂商。**粘下面这段：**

```
按 ai-cross 做一次真实的跨厂商派发，只做这一件事，不要做别的：
运行这条命令，把一个情感分类派给 OpenAI 家族的 codex（明确不同于 Qoder 可能在用的 Claude/Gemini）：

codex exec -m gpt-5.4 -c model_reasoning_effort="low" -s read-only --skip-git-repo-check "判断情感只回一个词：正面/负面/中性。句子：客服态度极差，问三次都没人理。"

然后报告：1) 命令是否成功执行、codex 返回了什么；2) 确认这是跨厂商派发（你/宿主 vs 被派方 OpenAI）；3) 一句话说明为什么交叉验证必须换厂商。命令若被拦截需要审批，如实说明你看到了什么。
```

> 若这台机器没装 codex，改测校验器（也是真命令执行，且顺带展示 ai-cross 的防降级能力）：
> ```
> 运行 python ai-cross/references/verify_model.py --provider "Zhipu GLM"，报告输出，并解释它为什么比"信 CLI 自报模型名"更可靠。
> ```

**关键**：命令返回后，别只信 Qoder 复述——让它把**实际执行的命令和原始输出**贴出来（我们核实真发生了，不采信自述，这本身就是 skill 的铁律）。

## 四、要抓的素材（文章用）

- 阶段 1 的**完整派发计划**截图（尤其"交叉审查必须显式换厂商"那条）。
- 阶段 2 里 Qoder **实际跑的命令 + 原始返回**截图，以及**审批弹窗**（如果有）——"命令被 sandbox/审批拦截"本身是很好的真实素材。
- 任何**没按预期**的地方照原样记下来（比如它漏了"排除内部通道"、或把交叉审查派给了同厂商）——反例和正例一样有价值。

## 五、判定

- 阶段 1 五问基本答对 → host-agnostic 的**决策层**在第三个宿主(Qoder)上也成立。
- 阶段 2 真跑通一次跨厂商命令 → **执行层**也成立。
- 两个都过 → 就有了"同一个 skill、Claude Code / Codex / Qoder 三个宿主都能正确路由"的三点证据，是文章 host-agnostic 最硬的一段。
