# 基准结果:5 臂 × 4 任务 × 3 轮(2026-07-08)

## 方法

- **臂**:codex-low(gpt-5.4-mini/low)、codex-high(gpt-5.5/xhigh)、glm-low(glm-5-turbo)、step-low(step-router-v1)、step-mid(step-3.5-flash)
- **仪器**:`codex exec --json` → `turn.completed.usage`;`claude -p --output-format json` → `usage`。**不用 codex 打印的 `tokens used`**(它是未命中缓存的计费部分,随缓存在 188~19,538 间跳)
- **verifier**:程序化判定,经 9 项正/负样本自检(含 CLI 横幅噪声、修改入参、靠 `range(step=0)` 副作用蒙混)。代码类任务**剥掉模型自带的自测块,跑我们自己的用例**
- **控制**:每 trial 注入唯一 nonce;每 trial 独立进程
- **排除**:glm-mid(glm-5.1)因持续 529 过载被移出,单次卡 190s

## 站得住的三条结论

1. **厂商差异 >> 档位差异(本任务集内)**
   StepFun 两个档位都只有 33%(计数系统性偏大:报 17/18,实为 16);codex 与 GLM 各档位全 100%。
   与另一次统计任务中 StepFun「方法标注错误 + 两处算术错误」**独立互证**。
   → **路由时"哪家"比"哪档"更要紧;升档救不了系统性缺陷。**

2. **thrash 可测且真实**
   同样答对 recon-count:codex-low 用 **3 轮工具调用 / 81,824 input**;codex-high 用 **1 轮 / 44,826**。
   便宜档多花近 **1.8 倍上下文**才到达同一答案。
   (这是早先被撤回的"多烧 70% token"结论的正确版本——用稳定的 input 总量测。)

3. **延迟按厂商差一个数量级**
   codex 62–90s;GLM/StepFun 5–14s。延迟敏感场景是硬约束。

## 本数据【不能】支持的

- **不能**说"便宜档在中难度任务上不行"。两个实现任务上**全部 5 臂 100%**。
  早先"mini 失败"是**它自带的自测与自身实现矛盾**(交付物缺陷,n=1),与"实现正确性"是两个不同指标,当时被混为一谈。
- 因此**本任务集难度不足以区分档位**,不能验证路由表的"中难度→中档"边界。**需要更难的任务。**

## 过程中修掉的真 bug(比结果本身更有价值)

1. **`cc_switch.py` 把 API 错误当答案交付**:`claude -p --output-format json` 遇 529 时仍 exit 0,把 `"API Error: 529..."` 塞进 `result` 字段 → 已加 `is_error`/`api_error_status` 校验(exit 8)。
2. `subprocess.TimeoutExpired` 未捕获,直接抛 Python 堆栈 → exit 9 + 可操作提示。
3. verifier 把 codex 的 stderr 横幅当代码 `exec()` → 假阴性,差点把"便宜模型失败"记成结论。
4. verifier 的负样本靠 `range(step=0)` 抛 ValueError 的**副作用蒙混过关** → 补测 `n=-1`。
5. `gpt-5.3-codex` 型号在 ChatGPT 账号下**根本不存在**,而它曾是 channels.md 的高档默认。

## 方法学教训(本轮共犯错 4 次,全部由自检抓出)

1. 用 `tokens used` 做跨档比较 → 该数字随缓存跳 43 倍。
2. 据一次冷调用断言"GLM 端点零缓存" → 热调用 fresh 仅 3,064。
3. 修正后又断言"StepFun 几乎不缓存" → 热调用 cache_read 达 29,952。
4. 把"模型自带自测失败"当成"实现能力不足" → 两个不同指标。

**共同根因:单次测量当稳态;隐藏变量(缓存状态)未控制;指标定义未分清。**

**固化规则:先证明尺子是准的,再量任何东西。**
