# 高难度基准结果（2026-07-08，受控环境，可引用）

## 方法与控制

- **任务**：4 个**自造**高难度任务，规避训练集污染
  - `h-interval` 跨午夜时段语言（跨午夜拆段 + 相邻必须合并 + 零长度非法 + 严格格式，四陷阱交织）
  - `h-checksum` 自造校验和（含 `%` 对负数返回非负的语义陷阱）
  - `h-recur` 51 步精确递推（逼模型**亲自迭代**，不能估算）
  - `h-ambig` 规格含真实歧义（必须显式标注 `# AMBIGUITY:`）
- **臂**：codex-low(gpt-5.4-mini/low)、**codex-advisor**(low 执行 + high 顾问)、codex-high(gpt-5.5/xhigh)、glm-low(glm-5-turbo)、step-low(step-router-v1)
- **仪器**：`codex exec --json` → `turn.completed.usage`；`claude -p --output-format json` → `usage`。**绝不用 codex 打印的 `tokens used`**（随缓存跳 43 倍）
- **verifier**：13 项正/负样本自检通过（含 CLI 横幅噪声、markdown 围栏、代码后跟中文散文、靠 `range(step=0)` 副作用蒙混、擅自排序修改入参）。代码类任务**剥掉模型自带自测块，跑我们自己的用例**
- **控制**：每 trial 唯一 nonce；trial1=冷 / trial2+=热，分开统计；**prompt 经 stdin 传入**（argv 会在首个换行截断）
- **环境**：superpowers 已移除；`~/.codex/AGENTS.md` 保留（用户工程规范，属真实使用条件）

## 结论一（最重要）：区分「模型亲自算」与「模型写代码让机器算」

StepFun 在 **h-checksum 拿 100%**（写自造校验和算法的**代码**，算术由 Python 执行），
却在 **h-recur 只有 33%**（模型**亲自迭代 51 步**）。

三次**独立**命中同一模式：

| 观察 | 任务性质 | StepFun |
|---|---|---|
| h-checksum（本轮） | 写代码，机器算 | ✅ 100% |
| h-recur（本轮） | 亲自迭代 51 步 | ❌ 33% |
| recon-count（简单基准） | 肉眼数 assert | ❌ 33%（数成 17/18） |
| 统计任务（全力模式实测） | 亲自推导四分位 | ❌ 方法标注错 + 两处算术错 |

且 `step-mid` 同为 33% —— **升档救不了，这是能力维度缺失，不是档位不够。**

**路由规则（已写入 SKILL.md）**：
- 模型**亲自**计算/计数/推导 → 对通道质量**高度敏感**，避开有质量备注的通道，结果必须编排者独立核验
- 模型**写代码**、机器执行 → 不敏感，便宜档随便用
- **推论：需要精确数值时，宁可让模型写代码，也不要让它心算。** 既提高正确率，又让结果可被独立跑一遍验证。

## 结论二：advisor 模式的经济学在 CLI 架构下反转

| | codex-low | codex-advisor | 倍数 |
|---|---|---|---|
| 正确率 | 100% | 100% | **零增益** |
| input 中位 | 19.6k–39.4k | 41.7k–61.7k | **约 2.1×** |
| 耗时 | 56–63s | 121–172s | **约 2.2×** |

Anthropic 的 advisor 工具基准称成本**反降 11.9%**；我们实测**涨约 110%**。

**原因**：原生 advisor 是 Messages API 的 **in-request 工具，不重发上下文**；跨 CLI 咨询每次要付一次完整的 20k–30k 冷启动足迹。

**⚠️ 天花板效应（必须标注）**：codex-low 本身就 100% 正确、`tools=0` 零 thrash —— **顾问根本没有错可纠**。文章的成本收益来自"减少试错轮数"，我们的任务没产生试错，**故无法复现该收益，也无法证伪它**。

**能确定的**：执行者已成功时，advisor 是纯浪费（2.1×）。**只能挂在升级阶梯上——观察到 thrash/失败才咨询一次，绝不预防性调用。**

## 结论三（第三次）：档位区分不出来

最便宜的 `codex-low` 在**全部四个自造硬任务**上 100%，包括四陷阱交织的实现题与负数取模陷阱。
`codex-high` 只多烧约 2× 推理 token（out 838 vs 390 等），**零准确率增益**。

**故：默认往低档派，靠验证失败再升级。** 档位的作用**远小于**「通道质量 + 是否让模型亲自算」。

## 其他观察

- **延迟按厂商差一个数量级**：codex 56–74s；GLM/StepFun 7–78s（简单任务 7–14s）。
- **StepFun 话痨**：h-interval out=6,271、h-recur out=7,710，是 codex 的 15–30 倍。
- **缓存双峰确认**：所有端点热调用 fresh 大幅下降（step-low 30,462 → 125；glm-low 28,891 → 3,164）。

## 本数据【不能】支持的

- 不能给出任何"省 X%"的跨厂商成本结论（单价差数量级，订阅制无法换算美元）。
- n=3，方差仍大；仅 4 个任务，覆盖面窄。
- **自造任务只是缓解训练集污染，不能证明消除**。
- advisor 的准确率收益因天花板效应**未被检验**。

## 过程中修掉的关键 bug（比结果更值钱）

1. **多行 prompt 经 argv 传给 `codex exec` 会在首个换行截断**（Windows）。codex 只收到规格第一行，于是完全正确地回答"你没把算法贴出来"。这让整套高难度基准误判为 codex 全线 0%，且一度被错误归因为"目标 CLI 的技能污染"。**修复：走 stdin。**
2. verifier 把 CLI 横幅当代码 `exec()` → 假阴性；改为**渐进式编译**（找能编译且定义目标函数的最长前缀）。
3. verifier 负样本靠 `range(step=0)` 抛 ValueError 的**副作用蒙混过关** → 补测 `n=-1`；`items.sort()` 改入参检测不出 → 改用乱序入参。
4. `cc_switch.py` 把 API 错误（529）当答案交付（`claude -p --output-format json` 报错时仍 exit 0）→ 加 `is_error` 校验。

**方法学第 0 条：先证明尺子是准的。判定模型失败前，先 dump 出它实际收到的输入。**
